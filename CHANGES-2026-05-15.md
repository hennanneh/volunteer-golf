# Tournament-prep changes — 2026-05-15

End-of-day snapshot of everything wired up today, ahead of the ~2026-05-29 tournament.

## What changed (8 layers of data-loss safety + monitoring)

1. **Activity log made unbounded** — removed all 200/500-entry trims (server-side merge, PATCH handler, deployment log, startup log, client `logActivity`, client display + the "limited to 200" badge). `data.json` activity history now retained indefinitely.

2. **Staleness window tightened 30 → 10 min** — `STALE_READ_MAX_AGE_MS` in `server.js`. Saves from tabs older than 10 min now get HTTP 409 `STALE_READ`. Reduced silent-override window 3×.

3. **Backups bumped 60 → 5 min** — `backup.sh` cron changed from `0 * * * *` to `*/5 * * * *`. Same script (no-ops when nothing changed). Pushes to `git@github.com:hennanneh/golf.git`. Worst-case data-loss window ~5 min instead of ~60.

4. **ntfy push alerts (sanitized)** — public ntfy.sh, topic `golf-alerts-1c97adfdf03ef59f`. Counts-only initially, escalated to full PII per user decision. Topic name is now password-sensitive — never paste in screenshots / public messages.

5. **Per-event activity push** (`activity-log-watcher.sh`, PM2 `golf-activity-watcher`) — polls `data.json` every 10 s, pushes each new activity log entry as it lands. Title `"User — Action"`, body has Time / Type / target / details. Click opens colonialvolunteers.golf.

6. **Plain-English failure pushes** (`live-monitor.sh`, PM2 `golf-live-monitor`) — tails PM2 stderr, pushes blocked saves / failed logins / auth issues with friendly titles like "Save blocked — outdated browser tab" instead of "REJECTED stale dataReadAt". Includes initials/email-prefix hints (e.g. "JAS") so brute-force patterns are spottable.

7. **Server-side bulk-save diff logging** — `mergeVolunteerSave` in `server.js` now generates an activity log entry for every record the merge accepts, even when the SPA didn't call `logActivity`. Tagged `(via bulk save)`. Closes the silent-override audit gap. Entries carry `actorHole` + `targetHole` for cross-hole detection.

8. **Server-side enforcement (the "agent")** — `captainAuthorized()` in `server.js`. Captain role can only modify records where `existing.hole === incoming.hole === captain.hole`, plus their own profile. Hole reassignment is admin-only. Rejected attempts get `action: 'rejected-bulk-edit'` activity entries that the watcher pushes as 🛑 high-priority alerts. Admin / Chair / AsstChair retain full write authority.

9. **Hourly digest** (`hourly-digest.sh`, cron `5 * * * *`) — counts heartbeat. High priority if heaviest user > 50 saves/hr OR loginfail concentration > 10/hr.

## Push notification alphabet

| What you see | What it means | Action |
|---|---|---|
| `User — Login` (low) | Successful login | Routine — no action |
| `User — Schedule edit` (low) | Captain edited a schedule | Glance, ignore if normal |
| `User — Schedule edit (via bulk save)` (low) | Bulk-save accepted a change. Normal during save flows. | Glance — investigate if target hole doesn't match user's hole |
| `🚨 CROSS-HOLE: User → Target` (high) | Captain edited vol in another hole AND server allowed it (rare — would mean enforcement missed) | Investigate immediately |
| `🛑 AUTO-REJECTED: User → Target` (high) | Captain attempted cross-hole edit; server blocked it; **no data was lost** | Audit only — no data action needed; figure out why their browser tried |
| `Save blocked — outdated browser tab` (high) | Captain's tab >10 min old; their save was rejected | Tell them to reload |
| `Save blocked — old cached browser` (high) | Captain on pre-5/14 cached SPA; bulk POST refused | Tell them to hard-refresh (Ctrl+Shift+R) |
| `Login failed — JAS` (default) | Wrong password attempt | Watch for repeats from same hint = brute-force |
| `Unauthorized save attempt — MB` (default) | Logged-in user tried something they don't have permission for | Usually the broadcast-handler bug |
| `Old browser detected (legacy auth path)` (low) | Captain hasn't refreshed since auth update | Inform them when convenient |
| `Golf hourly digest` | Hourly heartbeat with totals | Glance |

## Tradeoffs accepted

- **PII in push payloads.** Names, target volunteer names, schedule changes are sent over public ntfy.sh. Topic name is password-sensitive. Acceptable for tournament prep; revisit later if needed (self-host ntfy or move to Twilio SMS).
- **Verbose alert volume.** Every activity log entry → push. ~25-100 pushes/day. Intentional for the next ~2 weeks to learn baseline; can dial back via `pm2 stop golf-activity-watcher` or threshold tuning.
- **No tail-end deduplication.** Bulk save by a captain who *also* logged the action client-side will produce two activity entries (one client-generated, one server-generated `(via bulk save)`). Acceptable noise; tells you both intent and what landed.

## How to dial back later (post-tournament)

```bash
# Stop the firehose; rely on hourly digest + failure pushes
/usr/local/bin/pm2 stop golf-activity-watcher

# Or stop both watchers and rely solely on hourly digest
/usr/local/bin/pm2 stop golf-activity-watcher golf-live-monitor

# Or change failure-alert thresholds — edit the script then restart
nano /root/volunteer-golf/live-monitor.sh
/usr/local/bin/pm2 restart golf-live-monitor
```

## Recovery procedure (if data ever needs restoring)

```bash
cd /root/volunteer-golf

# See snapshots
git log --oneline -- data.json | head -20

# Inspect a snapshot without restoring
git show <commit>:data.json | jq '.volunteers | length'

# Restore (replaces current data.json with chosen snapshot)
git show <commit>:data.json > data.json
/usr/local/bin/pm2 reload volunteer-golf
```

GitHub remote (`origin`) is the off-site backup if the droplet is lost.
